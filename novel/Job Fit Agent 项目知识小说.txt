Job Fit Agent 项目知识小说

这是一本用故事方式讲解本项目的学习笔记，适合从零复习项目结构、简历读取、技能匹配、Web 页面、AI 分析和面试讲法。

============================================================

# 第一章 项目醒来

一开始，文件夹像一张没整理过的书桌。

根目录里有 `app.py`、`tools.py`、`web_app/`、`examples/`、`tests/`，还有 `.venv`、`.idea`、`ui.skills`、`常用skills` 这样看起来让人心里发慌的名字。你盯着它们，第一反应不是“这是我的项目”，而是“这里到底哪些能动，哪些不能动”。

后来项目慢慢被分清了身份。

`app.py` 是命令行入口，像项目的前台接待。你在终端里运行 `python app.py`，它负责把简历、岗位 JD、评分结果、终端报告串起来。

`tools.py` 是真正的机器房。它负责读取文件、抽技能、算分、生成诊断。很多项目表面看起来是 AI，其实能不能稳定跑，靠的就是这种机器房里的规则逻辑。

`web_app/` 是网页版本。它让项目从“只能在命令行跑”变成“可以上传文件、点按钮、看结果”。这对简历项目很重要，因为面试官不一定愿意读一堆终端输出，但他很容易理解一个能打开的页面。

`examples/` 是演示数据。里面有示例 JD 和简历。没有它，别人拿到项目后不知道怎么试。

`tests/` 是安全绳。每次你改了读取、评分、页面逻辑，测试能告诉你核心功能有没有被改坏。

至于 `.venv` 和 `.idea`，它们不是项目主体。`.venv` 是本机 Python 环境，`.idea` 是编辑器配置。它们能留在本地，但不适合放进简历展示仓库。真正要展示的是代码、示例、说明和测试。

项目第一次像项目，是从目录被讲清楚开始的。

以前你看到一堆文件，会觉得它们都在同时向你说话。现在你知道，项目可以被分成几层：

- 输入层：简历文件、JD 文件、网页上传。
- 处理层：读取文本、抽取技能、匹配评分。
- 展示层：命令行报告、Web 页面、导出文件。
- 增强层：可选 AI 分析和汇总报告。
- 保障层：测试、`.gitignore`、README。

当你能说出这几层，这个项目就不再只是“别人帮我写出来的一堆代码”。它开始变成你能解释的东西。

## 这一章你要记住

一个项目不是靠文件多显得专业，而是靠职责清楚。

在 Job Fit Agent 里：

- `app.py`：命令行流程入口。
- `tools.py`：核心业务逻辑。
- `web_app/main.py`：Web 后端入口。
- `web_app/templates/`：网页 HTML 模板。
- `web_app/static/`：CSS 和 JS。
- `examples/`：演示输入。
- `tests/`：验证逻辑。

面试时不要一上来讲“我用了 AI”。先讲清楚项目链路：输入是什么，处理过程是什么，输出是什么，哪里用了规则，哪里可选调用模型。

# 第二章 简历进门

简历不是一开始就愿意乖乖进系统的。

有的简历是 `.docx`，像新式门卡，`python-docx` 能直接读。  
有的是 `.pdf`，像贴在墙上的海报，`pypdf` 要一页一页把文字抠出来。  
最麻烦的是 `.doc`，它像旧时代的钥匙，不能直接被现代工具轻松打开。

你最开始遇到的问题就是 `.doc` 没有识别到。明明最开始可以读，后来 Web 页面却挡住了它。那不是模型问题，也不是简历问题，而是程序里某一层把 `.doc` 拦在门外了。

后来门被重新打开。

在 `tools.py` 里，读取文件的总入口是 `load_text_file()`。它先看文件后缀：

- `.txt`、`.md`：直接当文本读。
- `.docx`：交给 `load_docx_file()`。
- `.pdf`：交给 `load_pdf_file()`。
- `.doc`：交给 `load_doc_file()`。

这就是“分发”的概念。一个总入口不需要自己干所有活，它只负责判断该把任务交给谁。

`.doc` 的处理更像绕路。程序会先找 LibreOffice 的 `soffice`，尝试把 `.doc` 转成 `.docx`。如果这条路失败，再尝试用 Windows 上的 Microsoft Word COM 自动化转换。转换成功后，还是回到熟悉的 `load_docx_file()` 读取。

这背后有一个重要工程思想：不要为了兼容旧格式，把后面的所有逻辑都搞复杂。把旧格式先转成新格式，再走同一套新格式读取流程。

Web 页面也有自己的入口。用户上传文件后，`web_app/main.py` 里的 `save_upload()` 会先把上传文件保存到临时目录，然后再调用 `load_text_file()`。也就是说，Web 和命令行虽然入口不同，但最终都汇入同一个核心读取函数。

这很重要。否则你会出现一种可怕的情况：命令行能读 `.doc`，网页不能读 `.doc`；或者网页修好了，命令行又坏了。共用核心函数，就是为了减少这种割裂。

后来你又加了“取消选择”按钮。这个功能看似小，其实也是输入层的一部分。用户选错文件时，不应该只能刷新页面重来。好的输入层不仅要能接收文件，还要允许用户反悔。

于是页面里的文件输入框旁边多了“取消选择”。JS 监听文件选择变化：有文件时显示按钮，点击按钮时把 input 的 value 清空。这里没有高深算法，但有真实产品感。

项目开始像一个工具，而不是一次性脚本。

## 这一章你要记住

文件读取链路大概是：

```text
用户上传/命令行传入文件
        ↓
判断文件后缀
        ↓
调用对应读取函数
        ↓
得到纯文本
        ↓
进入技能抽取和评分
```

重点知识点：

- 后缀判断是输入分发。
- `.doc` 不能简单等同于 `.docx`，通常需要转换。
- Web 和命令行最好复用同一套核心读取函数。
- 小交互，比如“取消选择”，也是工程能力的一部分，因为它让用户少走弯路。

# 第三章 技能树法庭

简历变成文本后，项目真正开始审理案件。

岗位 JD 坐在一边，说：“我需要 Python、FastAPI、Docker、Git，还希望你懂大模型 API、Prompt Engineering、RAG、向量数据库和工具调用。”

候选人的简历坐在另一边，说：“我写过 Python，用过 FastAPI，做过 Docker 部署，也接过大模型 API。”

系统不是拍脑袋判断谁更好，而是打开 `tools.py` 里的 `SKILL_CATALOG`。这像一部小小的技能法典。每个技能都有几个属性：

- 标准技能名，比如 `Python`。
- 别名，比如 `py`。
- 所属类别，比如 `编程语言`。
- 权重，比如 Python 比普通小技能更重要。

为什么要有别名？因为人不会总用同一个词写简历。

有人写“向量数据库”，有人写“embedding”，有人写“向量检索”。如果程序只认一个词，就会错过很多信息。所以项目把不同表达归一化到同一个技能上。归一化的意义是：不要被表面文字骗了，要把同类能力合并起来看。

`extract_candidate_skills()` 负责从简历里抽候选人技能。  
`extract_job_requirements()` 负责从 JD 里抽岗位要求。  
它们背后都会调用 `build_skill_profile()`。

这个函数会遍历技能词典，检查文本里是否出现了某个技能的别名。如果出现，就记录这个技能、命中的别名、类别、权重和证据片段。

证据片段很重要。它让结果可解释。不是系统说“你有 Python”，你就必须信；系统可以展示它在文本哪里看到了 Python。

抽完技能之后，真正的审判是 `compute_match_score()`。

它会把候选人技能和岗位要求做集合对比：

- 两边都有的，是已匹配技能。
- JD 有但候选人没有的，是缺失技能。
- 候选人有但 JD 没要求的，是额外技能。

然后用权重算分。不是所有技能都一样重要，比如 AI 应用岗位里，大模型 API、Prompt、RAG 的缺失，比少一个普通工具词更严重。

系统还会按类别算覆盖率。这样结果不只是一个总分，还能看出候选人到底是后端强、AI 应用弱，还是部署能力弱。

这就是你之前看到的“李四 100 分”“妖六 53 分”“张三 37 分”背后的逻辑。分数不是 AI 瞎编，而是本地规则先算出来的。AI 后来最多负责把话说得更像专业分析，不能随便推翻本地分数。

这个边界很重要。

如果系统完全靠 AI 判断，它可能写出一篇看似高级的文章，却把分数、排序、短板说乱。你之前不喜欢那种长篇排序报告，就是因为它偏离了规则结果。后来项目改成：本地规则决定结构，AI 只补充表达。

这才是更稳的工程设计。

## 这一章你要记住

匹配评分不是魔法，主要是这几步：

```text
技能词典
  ↓
从简历抽技能
  ↓
从 JD 抽要求
  ↓
已匹配 / 缺失 / 额外技能
  ↓
按权重计算分数
  ↓
生成优势、短板、建议
```

重点知识点：

- `SKILL_CATALOG` 是规则系统的核心。
- 别名匹配解决“同一个能力不同写法”的问题。
- 权重让关键技能更重要。
- 可解释评分比单纯 AI 判断更稳定。
- AI 可以润色和补充，但不应该乱改本地规则算出的核心排序。

# 第四章 网页有了呼吸

命令行像一间机器房，能跑，但不够亲近。

于是项目有了 Web 页面。

在 `web_app/main.py` 里，`FastAPI(title="Job Fit Agent Web")` 像是给这个网页应用挂上门牌。浏览器访问 `/` 时，后端返回一个 HTML 页面；点击“一键处理”提交表单时，请求会来到 `/analyze`。

Web 项目最基础的关系是：

```text
浏览器页面
  ↓ 提交表单
FastAPI 路由
  ↓ 调用核心逻辑
tools.py
  ↓ 返回结果
Jinja2 模板渲染页面
```

页面模板在 `web_app/templates/pages/index.html`。它不是普通静态 HTML，而是 Jinja2 模板。模板里可以写 `{% if results %}`、`{% for item in results %}` 这样的语法，意思是：如果后端给了结果，就把结果循环渲染成候选人卡片。

CSS 在 `web_app/static/css/app.css`。它决定页面好不好看，按钮、卡片、进度条、结果区域都在这里被塑形。你一开始说“把页面弄好看一些”，本质上就是让这个项目从“能用”走向“可展示”。

JS 在 `web_app/static/js/progress.js`。它负责页面的呼吸。

为什么说是呼吸？因为没有 JS，用户点按钮后只能等待，页面像卡住了。加了进度条以后，用户知道程序正在上传、读取、评分、导出。即使真实后端处理还没完成，前端也能给出节奏感。

后来你又提出两个很真实的体验问题。

第一个是：本地规则分析点一下要有提示。  
因为用户可能以为点了按钮就会调用 AI，但默认其实是本地规则。于是 JS 在提交表单时检查 `AI智能分析` 是否勾选。如果没勾选，就弹出确认框：本次只用本地规则，不调用 AI。

第二个是：JD 和简历文件导入后不能取消。  
这是文件输入框的常见痛点。原生浏览器文件选择框不够友好，于是页面旁边加了“取消选择”按钮。JS 做三件事：

1. 监听文件选择变化。
2. 有文件时显示按钮和文件数量。
3. 点击取消时清空 input。

这些细节不会写进算法论文里，但会让面试官感觉：你做的不是一次性练习，而是一个考虑用户操作的工具。

Web 页面还有下载区。结果会保存到 `web_app/处理结果/`，然后页面提供 JSON、Markdown、Word 和 Excel 下载链接。JSON 给程序看，Markdown 给人快速读，Word 更像正式报告，Excel 适合筛选排名。

当一个项目能上传、能分析、能展示、能下载，它就有了完整应用的样子。

## 这一章你要记住

Web 版核心结构：

- `web_app/main.py`：FastAPI 路由和后端逻辑。
- `templates/pages/index.html`：页面结构。
- `static/css/app.css`：样式。
- `static/js/progress.js`：交互和进度条。
- `处理结果/`：导出文件保存位置。

重点知识点：

- FastAPI 负责接收请求。
- Jinja2 负责把后端数据变成 HTML。
- CSS 负责视觉层。
- JS 负责用户交互。
- 进度条、取消选择、确认提示都属于产品体验，不是可有可无的小装饰。

# 第五章 AI 只在被邀请时说话

这个项目里，AI 不是一直站在台前。

这点很重要。

一开始你不希望没点 AI 的时候页面也说 AI。因为那会让人误解：明明只是本地规则分析，为什么页面一直在讲智能复核？所以项目后来定下规矩：默认只跑本地规则；只有用户勾选“AI智能分析”，或者点击“写汇总报告”，才调用模型。

这叫可控调用。

在 `web_app/main.py` 里，AI 配置由 `load_ai_config()` 读取。它会看 `.env` 和 `web_app/.env`，寻找这些东西：

- provider：使用 OpenAI 兼容接口，还是 Anthropic Messages 兼容接口。
- api key：密钥。
- base url：请求地址。
- model：模型名。

如果缺少配置，就不应该神秘失败，而应该告诉用户没有配置。真实项目里，错误提示比沉默失败重要得多。

模型调用有两条路。

OpenAI 兼容接口会请求：

```text
/chat/completions
```

Anthropic Messages 兼容接口会请求：

```text
/v1/messages
```

你之前遇到过 `429 Too Many Requests`，意思是请求太多，被服务商限流。又遇到过 `401 Unauthorized`，意思是鉴权失败，通常是 key、url、provider 或请求格式不匹配。

这些错误听起来吓人，但本质上都是 HTTP 世界的门禁信号：

- `401`：你是谁，门禁不认。
- `429`：你来太频繁，先等一等。
- `500`：对方服务内部出问题。
- `504`：等太久，超时了。

项目后来做了两种 AI 能力。

第一种是“AI智能分析”。它不是让 AI 写一篇长文重新排名，而是让 AI 按 JSON 格式返回每个候选人的补充评价。后端再把它合并到本地规则结果里。分数和等级仍然由本地规则决定。

第二种是“写汇总报告”。点击按钮后，网页不下载文件，而是在页面里生成一段候选人汇总分析。前端同时显示进度条，让用户知道模型还在生成。

这个功能背后有一个重要分工：

```text
本地规则：负责稳定、可解释、可复现
AI 模型：负责补充表达、总结、专业化语言
```

如果把所有判断都交给 AI，项目会显得花哨但不稳。  
如果完全不用 AI，它又少了一点大模型应用项目的亮点。  
所以最好的方式是：规则打底，AI 增强。

AI 只在被邀请时说话。它一说话，也要按项目定好的格式说。

## 这一章你要记住

AI 调用链路：

```text
用户勾选 AI 或点击汇总报告
        ↓
读取 web_app/.env
        ↓
选择 provider
        ↓
拼请求 URL 和 headers
        ↓
发送 prompt
        ↓
解析返回结果
        ↓
合并到页面展示
```

重点知识点：

- `.env` 用来放密钥和模型配置，不要提交真实 key。
- `401` 多半是鉴权配置问题。
- `429` 是请求频率或额度问题。
- 本地规则和 AI 要分工清楚。
- AI 输出最好要求结构化，否则容易写散、写偏。

# 第六章 把项目放进简历

项目快完成时，新的问题出现了。

不是代码跑不跑，而是它能不能被别人看懂。

一个能放进简历的项目，不能只在你电脑上能跑。它要像一间收拾过的房间：门口有说明，桌上有示例，抽屉里有测试，私人物品不要摆在正中央。

所以你开始整理文件夹。

`.venv` 是本地环境，不该进仓库。  
`.idea` 是编辑器配置，不该作为项目主体。  
`web_app/处理结果/` 是运行时生成的报告，可以本地保留，但不适合全部提交。  
`dev_resources/` 和 `常用skills` 是辅助资料，不属于 Job Fit Agent 的核心功能。  
真正适合展示的是：

- `app.py`
- `tools.py`
- `prompts.py`
- `web_app/`
- `examples/`
- `tests/`
- `requirements.txt`
- `README.md`

`.gitignore` 就像门口的筛子。它告诉 Git：哪些东西不要带走。密钥、缓存、导出结果、本地环境都应该被忽略。这样以后你把项目发给别人，不会把自己的 key、临时文件和一堆无关资源也打包出去。

README 是项目的自我介绍。它要回答几个问题：

1. 这个项目解决什么问题？
2. 用了什么技术？
3. 怎么运行？
4. 有哪些功能？
5. 面试时怎么讲？

你基础差，所以不要在简历上写得太虚。不要写“精通 AI Agent 全链路架构”。更稳的说法是：

> 做了一个简历与岗位 JD 匹配分析系统，支持多格式简历解析、技能抽取、规则评分、Web 上传、报告导出和可选 AI 补充分析。

这句话不夸张，但信息量足够。

面试官如果追问“你项目里 AI 具体做了什么”，你可以说：

> 核心评分不是完全交给 AI，而是本地规则先算出匹配分、已匹配技能和缺失技能。AI 只在用户勾选后补充候选人评价，或者生成汇总报告。这样可以保留规则评分的稳定性和可解释性。

面试官如果追问“你怎么读 `.doc`”，你可以说：

> `.docx` 用 python-docx，PDF 用 pypdf，旧版 `.doc` 会先尝试用 LibreOffice 转换为 `.docx`，失败时再尝试 Word COM。转换成功后复用 `.docx` 读取逻辑。

面试官如果追问“Web 怎么做的”，你可以说：

> 后端用 FastAPI 接收上传文件和表单，Jinja2 渲染结果页，CSS 负责页面样式，JS 负责进度条、文件取消选择和汇总报告生成交互。

面试官如果追问“你怎么保证没改坏”，你可以说：

> 项目有 `tests/`，覆盖技能抽取、评分、路径解析、PDF 读取等核心逻辑。每次改完会跑 `python -m unittest discover -s tests -v`。
